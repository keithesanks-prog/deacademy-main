Mastering Spark Plans & explain(True)

Rob: Think of explain(True) like a grocery receipt.

It doesn't just show you "Eggs: $5.00". It shows you how the eggs were sourced from the farm (FileScan), transported by truck (Exchange), and placed on the shelf (Sort).

If you don't read the receipt, you might not realize you're paying for express shipping (Shuffle) on every single item!

You don't need to learn this as a "language". You just use it as a diagnostic tool (like an X-Ray) to see why your job is slow.

1. The 4 Plan Types (The Factory Line)

Spark doesn't just run your code. It optimizes it in 4 stages.

üìù

1. Parsed Logical Plan

"The Syntax Check"
Spark checks if your SQL/Code grammar is correct. It doesn't check if the tables actually exist yet.

üîé

2. Analyzed Logical Plan

"The Catalog Check"
Spark looks up the columns and tables in the Catalog. If you misspelled "customerID", this is where it fails.

‚ú®

3. Optimized Logical Plan

"The Magic" (Catalyst Optimizer)
Spark rewrites your query for speed. It reorders joins, pushes filters down (Predicate Pushdown), and removes unused columns.

üë∑

4. Physical Plan

"The Instructions"
The actual blueprint sent to Executors. This shows FileScans, Shuffles (Exchanges), and HashAggregates.

2. Narrow vs. Wide (The Speed Test)

The main thing that kills performance is movement. Does data stay put, or does it travel?

Feature Narrow Transformation üèéÔ∏è Wide Transformation üê¢
Examples filter, select, withColumn join, groupBy`, repartition
Data Movement None. Stays on the same machine. Shuffle. Moves across the network.
Plan Keyword Filter, Project Exchange (The Red Flag!)

3. Deep Dive: Reading the Plan

üîÄ
Repartition vs Coalesce

Repartition(24): Shows Exchange. Full Shuffle. Good for balancing.

Coalesce(5): No Shuffle. Merges partitions. Fast.

ü§ù
Join Strategies

BroadcastJoin: Fast. Sends small table to all nodes.

SortMergeJoin: Slow. Shuffles and Sorts both tables.

4. Real World Examples (How to Read It)

Let's look at real "receipts" printed by Spark and decode them.

Example 1: The Simple Filter (Narrow)

Code: df.filter(df.id > 100)

== Physical Plan ==
*(1) Filter (isnotnull(id#0L) AND (id#0L > 100))
+- *(1) Scan ExistingRDD[id#0L,name#1]

üïµÔ∏è Interpretation:
Look for the *(1) asterisk. That means "Whole Stage Codegen" (Fast!).
There is no Exchange keyword. This means data stayed on the same machine. This is a fast, Narrow transformation.

Example 2: The GroupBy (Wide)

Code: df.groupBy("dept").count()

== Physical Plan ==
*(2) HashAggregate(keys=[dept#0], functions=[count(1)])
+- Exchange hashpartitioning(dept#0, 200)
   +- *(1) HashAggregate(keys=[dept#0], functions=[partial_count(1)])
      +- *(1) Scan ExistingRDD[dept#0]

üïµÔ∏è Interpretation:
We see Exchange hashpartitioning.
This is the smoking gun! Spark had to Shuffle data based on the "dept" column to group it. The number `200` is the default partition count for shuffles.

Example 3: SortMergeJoin (The Heavy Lifter)

Code: df1.join(df2, "id") (Large Tables)

== Physical Plan ==
*(3) SortMergeJoin [id#0L], [id#6L], Inner
:- *(1) Sort [id#0L ASC NULLS FIRST], false, 0
: +- Exchange hashpartitioning(id#0L, 200)
+- *(2) Sort [id#6L ASC NULLS FIRST], false, 0
   +- Exchange hashpartitioning(id#6L, 200)

üïµÔ∏è Interpretation:
This is the classic "expensive" join.
1. It sees two Exchanges (Both tables are shuffled).
2. It sees two Sorts (Both tables are sorted by ID).
3. Finally, it does the SortMergeJoin.
If you see this for small tables, consider using broadcast(small_df) to speed it up!

Cheat Sheet: Keywords to Watch

Scroll to the bottom of explain(True) and look for these:

  • FileScan: Reading data from disk.
  • Exchange: SHUFFLE. Moving data over network. (Costly!)
  • HashAggregate: Doing math (Sum, Count) inside a GroupBy.
  • BroadcastExchange: Preparing a small table for a fast join.
