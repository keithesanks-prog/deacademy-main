Git & GitHub

Amazon Web Services

Infrastructure
üí° Analogy: The Land & Electricity

AWS provides the raw materials to build your factory. You rent the land (VPC), the storage warehouse (S3), and the servers (EC2).

  • S3: Infinite hard drive (Data Lake).
  • EC2: Rented computers.
  • IAM: Security guards (Who can enter?).

Spark & Databricks

Compute Engine
üí° Analogy: The Heavy Crusher Team

Spark is a way to use 100 computers at once to process massive data (Petabytes). It's the "engine".

Databricks is the premium garage that hosts Spark for you. It adds a nice UI, notebooks (like Jupyter), and management tools so you don't have to configure 100 servers manually.

Snowflake

Data Warehouse
üí° Analogy: The Organized Showroom

After Spark crushes the raw rocks (Data), you store the refined gold here.

Uniquely, Snowflake separates Storage (keeping data) from Compute (querying data). You can turn off the "Compute" machines to save money while still keeping the data.

SELECT * FROM sales_gold WHERE amount > 1000; -- Runs instantly on scaled compute

üîß The Assembly Line (dbt)

dbt (Data Build Tool) is where the magic happens. It runs inside your warehouse (Pulling data from Snowflake, transforming it, and saving it back to Snowflake).

dbt

dbt Fundamentals

Transformation
üí° Analogy: The Recipe Book

Instead of writing complex Python scripts to transform data, you just write SELECT statements.

dbt wraps your SELECT statement in a CREATE TABLE AS... command automatically. It handles dependencies (Table A must be built before Table B).

Jinja & Macros

Templating
üí° Analogy: SQL with Superpowers

SQL is static. Jinja lets you add if statements, for loops, and variables into your SQL.

-- A 'Macro' is a reusable function {% raw %} SELECT order_id, {% for payment_method in ['card', 'cash'] %} sum(case when method = '{{payment_method}}' then amount end) as {{payment_method}}_amount, {% endfor %} FROM orders {% endraw %}

Materializations

Config
üí° Analogy: How do you want it delivered?

1. View (Default)

Virtual table. Runs the complete query every time you read it. Fresh data, but slow.

2. Table

Physical file. Fast to read, but you have to rebuild the WHOLE thing to update it.

3. Incremental

The smartest one. Only adds new rows since the last run. Complex to set up, but very fast.

Apache Airflow

Orchestration
üí° Analogy: The Shift Manager

Someone needs to tell the robots when to start.

Airflow uses Python to define a DAG (Directed Acyclic Graph). It says:

  1. "At 8:00 AM, tell Spark to ingest data."
  2. "Once Spark is done, tell dbt to transform it."
  3. "If dbt fails, send an email to the Engineer."
