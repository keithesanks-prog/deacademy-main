Configuration

Show the working tree status (modified, staged, untracked files)

Commit staged changes with a descriptive message

üîê Authentication & Credentials

Identity vs Authentication

This is the #1 confusion in Git!

üìõ Identity (Name Tag)

Who you claim to be in commits.

git config user.name

Like a sticker on your shirt. You can write "Batman" and Git will believe you.

üîë Authentication (Key Card)

Your actual login credentials.

git push

The password/token checked by GitHub to allow access.

1. Managing Identity (The Name Tag)

This sets the name and email that appear in your commit history (`git log`). It DOES NOT log you in.

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
Set your global identity for all repos

2. Managing Authentication (The Login)

If Git isn't asking for a password, your credentials are likely cached. To switch accounts on Windows, you must clear this cache.

Updating GitHub Login on Windows
  • Open the Start Menu and search for "Credential Manager"
  • Select Windows Credentials (not Web Credentials)
  • Scroll down to find git:https://github.com
  • Click the arrow to expand, then click Remove (or Edit)
  • Try to git push again - Git will now prompt you for the new account!
Pro Tip: CLI Method
You can quickly delete cached credentials using the command line:
C:\> cmdkey /delete:git:https://github.com

3. Personal Access Tokens (PAT)

GitHub no longer accepts account passwords for Git over HTTPS. You must use a Personal Access Token or try SSH.

Generate a Token
1. Go to GitHub Settings > Developer Settings
2. Select "Personal access tokens (Tokens (classic))"
3. Generate new token with 'repo' scope
4. Copy it immediately!
Use the Token
When Git prompts for a password, paste your PAT instead of your account password.

4. SSH vs HTTPS

HTTPS

Pros: Easiest to set up, works everywhere.

Cons: Requires re-entering token if cache expires.

https://github.com/user/repo.git
SSH

Pros: Set up once, never type password again.

Cons: Initial setup (keys) is more complex.

git@github.com:user/repo.git

üåø Branching & Merging

Branch Management

git branch
List all local branches (* indicates current branch)
git branch <branch-name>
Create a new branch
git checkout <branch-name>
Switch to an existing branch
git checkout -b <branch-name>
Create and switch to a new branch in one command
git switch <branch-name>
Modern way to switch branches (Git 2.23+)
git switch -c <branch-name>
Create and switch to a new branch (modern syntax)
git branch -d <branch-name>
Delete a branch (safe - prevents deleting unmerged branches)
git branch -D <branch-name>
Force delete a branch (even if unmerged)
Example: Feature Branch Workflow
$ git checkout -b feature/user-login
Switched to a new branch 'feature/user-login'
$ # Make changes to files
$ git add .
$ git commit -m "Implement user login functionality"
[feature/user-login abc123] Implement user login functionality

Merging

git merge <branch-name>
Merge specified branch into current branch
git merge --no-ff <branch-name>
Merge with a merge commit (preserves branch history)
git merge --squash <branch-name>
Combine all commits from branch into one commit
Example: Successful Fast-Forward Merge
$ git checkout main
Switched to branch 'main'
$ git merge feature/new-button
Updating abc123..def456
Fast-forward
button.js | 25 +++++++++++++++++++++++++
styles.css | 10 ++++++++++
2 files changed, 35 insertions(+)
create mode 100644 button.js
Example: Merge with Conflicts
$ git merge feature/update-header

Automatic merge failed; fix conflicts and then commit the result.
# Step 1: Check status
$ git status

# Step 2: Open index.html and you'll see conflict markers:
<<<<<<< HEAD
<h1>Welcome to Our Site</h1>
=======

>>>>>>> feature/update-header
# Step 3: Edit file to resolve (choose one or combine)

# Step 4: Mark as resolved
$ git add index.html
# Step 5: Complete the merge
$ git commit
[main xyz999] Merge branch 'feature/update-header'
Merge Conflicts
When Git can't automatically merge changes, you'll need to manually resolve conflicts. Edit the conflicted files, then use git add and git commit to complete the merge.

Rebasing

git rebase <branch-name>
Reapply commits on top of another branch
git rebase -i HEAD~3
Interactive rebase of last 3 commits (edit, squash, reorder)
Example: Rebasing Feature Branch onto Main
# Step 1: Check current branch and commits
$ git log --oneline --graph -5
* def456 (HEAD -> feature/api) Add API endpoint
* abc123 Setup API structure
* 9876543 (main) Update README

$ git fetch origin main

$ git rebase origin/main

# Step 4: View the new linear history

* xyz888 Setup API structure

* 9876543 Update README

Merge vs Rebase
Merge: Preserves complete history, creates merge commits
Rebase: Creates linear history, rewrites commit history (don't rebase public branches!)

üåê Remote Repositories

Remote Management

git remote
List all remote repositories
git remote -v
List remotes with URLs (fetch and push)
git remote add <name> <url>
Add a new remote repository
git remote remove <name>
Remove a remote repository
Example: Adding a Remote
$ git remote add origin https://github.com/username/repo.git
$ git remote -v
origin https://github.com/username/repo.git (fetch)
origin https://github.com/username/repo.git (push)

Syncing with Remote

git fetch
Download objects and refs from remote (doesn't merge)
git fetch <remote>
Fetch from a specific remote
git pull
Fetch and merge changes from remote branch (fetch + merge)
git pull --rebase
Fetch and rebase local commits on top of remote changes
git push
Push commits to remote repository
git push -u origin <branch-name>
Push branch and set upstream tracking
git push --force
Force push (overwrites remote history - use with caution!)
Force Push Warning
Never use --force on shared branches! It rewrites history and can cause issues for other developers. Use --force-with-lease as a safer alternative.

Branch Tracking

git branch -r
List remote branches
git branch -a
List all branches (local and remote)
git checkout --track origin/<branch-name>
Create local branch tracking remote branch

Undoing Changes

git reset <file>
Unstage a file (keep changes in working directory)
git reset --soft HEAD~1
Undo last commit, keep changes staged
git reset --mixed HEAD~1
Undo last commit, keep changes unstaged (default)
git reset --hard HEAD~1
Undo last commit and discard all changes
git revert <commit-hash>
Create new commit that undoes changes from specified commit
git checkout -- <file>
Discard changes in working directory for a file
git restore <file>
Modern way to discard changes (Git 2.23+)
git restore --staged <file>
Unstage a file (modern syntax)
reset --soft
Moves HEAD, keeps staging & working directory
reset --mixed
Moves HEAD, resets staging, keeps working directory
reset --hard
Moves HEAD, resets staging & working directory

Stashing

git stash
Save current changes temporarily and revert to clean working directory
git stash save "description"
Stash with a descriptive message
git stash list
List all stashed changes
git stash pop
Apply most recent stash and remove it from stash list
git stash apply
Apply most recent stash but keep it in stash list
git stash drop
Delete most recent stash
Example: Stash Workflow
$ git stash save "WIP: user profile feature"
Saved working directory and index state On main: WIP: user profile feature
$ git checkout hotfix-branch
$ # Fix the bug...
$ git checkout main
$ git stash pop
On branch main, changes to be committed...

Cherry-Picking

git cherry-pick <commit-hash>
Apply changes from a specific commit to current branch
git cherry-pick <hash1> <hash2>
Cherry-pick multiple commits

Squashing Commits

Squashing combines multiple commits into a single commit, creating a cleaner history. Useful for consolidating work-in-progress commits before merging.

Interactive Rebase Commands
pick = keep commit as-is
squash (or s) = combine with previous commit, keep both messages
fixup (or f) = combine with previous commit, discard this message
reword = keep commit but edit message
drop = remove commit entirely

üîÄ GitHub Squash and Merge

When merging a pull request on GitHub, you can squash all commits into one clean commit on the target branch.

GitHub Squash Merge Workflow
  • Create feature branch and make multiple commits locally
  • Push branch to GitHub: git push -u origin feature/my-feature
  • Open a Pull Request on GitHub
  • Click the dropdown arrow next to "Merge pull request"
  • Select "Squash and merge"
  • Edit the commit message (GitHub combines all commit messages by default)
  • Click "Confirm squash and merge"
‚úÖ When to Squash
‚Ä¢ Multiple WIP commits
‚Ä¢ "Fix typo" commits
‚Ä¢ Experimental changes
‚Ä¢ Before merging to main
‚ùå When NOT to Squash
‚Ä¢ Distinct features in one PR
‚Ä¢ Important commit history
‚Ä¢ Already pushed to shared branch
‚Ä¢ Commits from multiple authors
Example: Before vs After Squashing
BEFORE (messy history):
abc123 Add user model
def456 Fix typo in user model
ghi789 Add validation
jkl012 Fix validation bug
mno345 Update tests
pqr678 Fix test typo
AFTER (clean history):
xyz999 Implement user authentication with validation and tests
Squashing Warning
Only squash commits that haven't been pushed to a shared branch! If you've already pushed, squashing requires force-pushing which can disrupt other developers. Use git push --force-with-lease if you must force-push.

üîÑ Common Workflows

Feature Branch Workflow
  • Create a new branch: git checkout -b feature/new-feature
  • Make changes and commit: git add . && git commit -m "Add new feature"
  • Push to remote: git push -u origin feature/new-feature
  • Create pull request on GitHub/GitLab
  • After review, merge into main branch
  • Delete feature branch: git branch -d feature/new-feature
Hotfix Workflow
  • Create hotfix branch from main: git checkout -b hotfix/critical-bug main
  • Fix the bug and commit: git commit -am "Fix critical bug"
  • Merge into main: git checkout main && git merge hotfix/critical-bug
  • Tag the release: git tag -a v1.0.1 -m "Hotfix release"
  • Merge into develop: git checkout develop && git merge hotfix/critical-bug
  • Delete hotfix branch: git branch -d hotfix/critical-bug
Syncing Fork with Upstream
  • Add upstream remote: git remote add upstream <original-repo-url>
  • Fetch upstream changes: git fetch upstream
  • Checkout your main branch: git checkout main
  • Merge upstream changes: git merge upstream/main
  • Push to your fork: git push origin main
Resolving Merge Conflicts
  • Attempt merge: git merge feature-branch
  • Check conflicted files: git status
  • Open conflicted files and resolve conflicts (remove <<<, ===, >>> markers)
  • Stage resolved files: git add <resolved-files>
  • Complete the merge: git commit
