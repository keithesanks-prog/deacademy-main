The Analogy: The Magic Photocopier

Imagine a photocopier that works instantly, uses no paper, and no ink.

It doesn't physically copy the document. It just gives you a new "access card" (Pointer) to the original document.

üîß How it Works: Metadata Magic

According to the blueprint (and science):

  • Snapshot: Quickly takes a snapshot of any Table, Schema, or Database.
  • Shared Brain: The Clone and the Original share the SAME Micro-partitions initially.
  • Zero Storage Cost: You don't pay for the shared data. You only pay for CHANGES.
  • Independent: Modifying the clone (e.g. for Dev/Test) helps you verify logic without breaking Production.

üß™ Lab Phase 1: The Instant Copy

Let's spin up a full Test environment in 2 seconds.

Step 1: The Magic Command
-- Clone the entire Production Database
CREATE DATABASE test_db
CLONE prod_db;

> Database TEST_DB successfully created.

Rob: That's it? Yes.
It copied Pointers, not data. Even if `prod_db` was 100TB, this finishes instantly.
You can do this for Tables and Schemas too!

üíî The Breakup: When Data Changes (Post-DML)

The clone and original remain friends (share storage) until one of them changes. This is called Copy-on-Write.

We updated Table A2. Snowflake didn't touch MP D (because A1 needs it!). Instead, it wrote the changes to a NEW Micro-partition (MP E) and pointed A2 to it.

üß™ Lab Phase 2: The Breakup

Now let's verify that they are truly independent (Copy-on-Write).

Step 1: Modify the Clone
-- Update data in the TEST environment
UPDATE test_db.public.my_table
SET name = 'Clone_Modified'
WHERE id = 1;
Step 2: Verify Total Independence
-- Check the Original PROD table (Should be unchanged)
SELECT * FROM prod_db.public.my_table;
-- Result: 'Original_Name' (Untouched!)

-- Check the CLONE (Updated)
SELECT * FROM test_db.public.my_table;
-- Result: 'Clone_Modified'
Rob: See? You can destroy the Test DB and Prod won't feel a thing.
And you only paid for the storage of that one changed micro-partition!
