üìñ What is DynamoDB?

DynamoDB is a serverless, NoSQL, key-value database designed for high performance at massive scale. It doesn't use tables with rigid columns like SQL; instead, it stores JSON-like items.

üß• Visual Analogy: The Coat Check (vs The Library)

SQL Database = A Library

  • You can ask complex questions: "Find me all books about 'History' written in '1995' that are 'Red'."
  • The librarian has to walk around and search (Scans). It gets slower as you add more books.

DynamoDB = A Coat Check

  • You give a ticket number (Primary Key).
  • The attendant grabs exactly that coat instantly.
  • It doesn't matter if there are 10 coats or 10 million coats. The speed is purely O(1).
  • Constraint: You generally CANNOT say "Give me the beige trench coat" without the ticket number.

‚≠ê Key Features

  • 1. Scalability: Can handle millions of requests per second. Suitable for apps with varying traffic.
  • 2. High Availability: Built-in replication across multiple Availability Zones (AZs) for durability.
  • 3. Managed Service: Zero operational overhead. AWS handles provisioning, patching, and backups.
  • 4. Flexible Data Model: Supports both Key-Value and Document (JSON) models.
  • 5. Cost Savings: Serverless + Auto-Scaling means you only pay for what you use (no idle servers).
  • 6. Security: Fine-grained control via IAM (can restrict access down to the attribute level).

üîë The Primary Key (The Ticket)

This is the most critical concept. You must design your table around how you will access data.

PARTITION KEY (PK) - Mandatory

The unique ID that tells DynamoDB where to store the data.

Example: UserID = 101

SORT KEY (SK) - Optional

Allows you to sort multiple items with the same Partition Key.

Example: Timestamp = 2023-01-01

Combined (PK + SK): "Get User 101's data from Jan 2023."

‚öñÔ∏è Capacity Modes (Pricing)

On-Demand Mode

Auto-Scaling. Pay for what you use.

Best for: Unpredictable traffic, new apps.

Cost: ~$1.25 per million writes.

Provisioned Mode

Manual Scaling. You reserve X requests/sec.

Best for: Steady traffic (Pre-paid electricity).

Cost: Much cheaper if utilization is high.

üõë Limitations & Gotchas

  • 1. Expensive at Scale: DynamoDB can be more expensive than RDS/Aurora if not optimized. High throughput (Write Capacity) costs add up quickly.
  • 2. NoSQL Database (Not Relational): Bad choice for complex relationships. No JOINs. You cannot "Relate" tables easily.
  • 3. Complex Queries: Poor optimization for ad-hoc queries. aggregation (SUM, AVG) is hard. You can't easily do "SELECT * WHERE age > 25 AND city = 'NY'" without an index.

üß™ Interactive Lab: The Leaderboard

Let's build a Gaming Leaderboard.

1. Create Table (Boto3)

import boto3
dynamodb = boto3.resource('dynamodb')

table = dynamodb.create_table(
    TableName='GamingLeaderboard',
    KeySchema=[
        {'AttributeName': 'GameID', 'KeyType': 'HASH'},  # Partition Key
        {'AttributeName': 'Username', 'KeyType': 'RANGE'} # Sort Key
    ],
    AttributeDefinitions=[
        {'AttributeName': 'GameID', 'AttributeType': 'S'},
        {'AttributeName': 'Username', 'AttributeType': 'S'}
    ],
    BillingMode='PAY_PER_REQUEST'
)
print("Creating table...")
table.wait_until_exists()
print("‚úÖ Table Created!")

2. Add Score (PutItem)

table.put_item(
   Item={
       'GameID': 'Fortnite',
       'Username': 'Ninja',
       'Score': 9999,
       'Region': 'NA-East'
   }
)
print("‚úÖ Score added!")

3. Get Score (GetItem)

response = table.get_item(
    Key={
        'GameID': 'Fortnite',
        'Username': 'Ninja'
    }
)
print(f"Ninja's Score: {response['Item']['Score']}")
