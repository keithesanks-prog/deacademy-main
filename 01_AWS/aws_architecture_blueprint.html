Translating "Business Needs" into "Architecture"

üöÄ The Scenario: "Food Delivery App"

Imagine we are building UberEats. We have complex requirements. Let's break down how the Business Need drives the Service Selection.

Business Requirement (The "What") AWS Tool (The "How") Why this tool?
"We need a Mobile App that works securely for 1M users." API Gateway The "Front Door". Handles Auth (Cognito), Throttling (don't crash), and Routing to backend.
"We need to fetch Menus and Process Orders (Business Logic)." ECS / Lambda The "Kitchen". Runs your code (Python/Node). ECS for long-running servers, Lambda for quick tasks.
"If the kitchen system is rebooting, DO NOT LOSE the Order." SQS The "Buffer". Stores the order in a durable queue until the system is ready to process it.
"When an order is confirmed, notify the User, the Driver, and the Restaurant simultaneously." SNS The "Megaphone" (Fanout). Publishes 1 message -> Triggers 3 different actions (Push Notification, SMS, Email).
"We need to see where the driver is in REAL-TIME." Kinesis The "Conveyor Belt". Handles massive streams of GPS coordinates (100k updates/sec) without crashing.
"We need to store User Profiles and active Order Status instantly." DynamoDB The "Coat Check". Key-Value store. Instant O(1) access to "Order #123".
"The marketing team wants to analyze last month's sales trends." Athena The "Librarian". Runs SQL queries on archived logs/data in S3 without needing a database server.
