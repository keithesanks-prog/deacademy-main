üåü Star Schema - Dimensional Modeling

What is a Star Schema?

A star schema organizes data into:
‚Ä¢ Fact Table (center) - Measurements, transactions
‚Ä¢ Dimension Tables (points) - Context, descriptions

Analogy: Think of a receipt
‚Ä¢ Fact = The receipt itself (what was bought, how much)
‚Ä¢ Dimensions = Details (who, what, when, where)

üìä Fact Table

  • What: Measurements
  • Contains: Foreign keys + metrics
  • Size: Millions/billions of rows
  • Example: Sales, orders
  • Think: The receipt

üè∑Ô∏è Dimension Tables

  • What: Context
  • Contains: Descriptive attributes
  • Size: Thousands/millions of rows
  • Example: Products, customers
  • Think: The details

üö® Detecting Fake Reviews - Yelp Fraud Detection

üö® The Problem: Yelp Fake Reviews

Yelp is facing an issue with fake reviews. The goal is to find:
1Ô∏è‚É£ Reviewers who are fake
2Ô∏è‚É£ Restaurants that hired fake reviewers

The Challenge: How do we design tables to capture the data we need?

ü§î The Thinking Process: From Problem to Tables

Step 1: What behaviors indicate fraud?
‚Ä¢ Posting too many reviews too fast
‚Ä¢ Always giving 5-stars or always 1-stars
‚Ä¢ Multiple accounts from same device
‚Ä¢ Copy-paste review text

Step 2: What data do we need to detect those behaviors?
‚Ä¢ Timestamps (to calculate velocity)
‚Ä¢ Device/IP info (to find duplicate accounts)
‚Ä¢ Review text features (to detect copy-paste)
‚Ä¢ Account metadata (to check profile completeness)

Step 3: How do we organize that data into tables?
‚Ä¢ Session data ‚Üí fact_review_session
‚Ä¢ User profile data ‚Üí dim_user_metadata
‚Ä¢ Text analysis ‚Üí fact_review_text_features
‚Ä¢ Timing data ‚Üí Enhance fact_review

üö© Pattern 1: Review Velocity

Behavior: Posting 10+ reviews/day

Data Needed:

  • Review timestamps (not just dates)
  • Time between reviews
  • Session duration

Table Design:

  • Add review_timestamp to fact_review
  • Add session_duration_sec to track how long user spent

üè™ Pattern 2: Browser Fingerprinting

Behavior: Multiple accounts from same device/browser

Data Needed:

  • IP address (hashed for privacy)
  • User agent (browser/OS/device)
  • Screen resolution (1920x1080)
  • Timezone (America/Los_Angeles)
  • Language (en-US)

Table Design:

  • Create fact_review_session
  • Link to fact_review via review_id
  • Note: Combination of fields creates "fingerprint"

üï∏Ô∏è Pattern 3: Account Completeness

Behavior: Fake accounts have no profile photo, 0 friends

Data Needed:

  • Account creation date
  • Days until first review
  • Profile photo uploaded?
  • Friend count

Table Design:

  • Create dim_user_metadata
  • Separate from dim_users (SCD Type 1 vs Type 2)

üìã How to Decide Table Names & Column Organization

Decision 1: Fact vs Dimension?

Ask: Does this table represent an event or a thing?

  • Event (Fact): Something that happened ‚Üí fact_review, fact_review_session
  • Thing (Dimension): An entity with attributes ‚Üí dim_users, dim_restaurant

Example: "A review session" is an event ‚Üí fact_review_session

Decision 2: What's the grain (one row = ?)

Ask: What does one row represent?

  • fact_review ‚Üí One row = One review
  • fact_review_session ‚Üí One row = One review session (device/IP/timing)
  • dim_user_metadata ‚Üí One row = One user's profile data

Why separate tables? Different grain! A review can have session data, but session data doesn't repeat for every review.

Decision 3: Which columns go in which table?

Ask: Does this column change at the same rate as other columns?

Column Changes When? Table
review_timestamp Each review fact_review
ip_address_hash Each session fact_review_session
friends_count User makes/loses friends dim_user_metadata
text_hash Each review (text analysis) fact_review_text_features

Decision 4: Should I create a new table or add columns?

Create NEW table if:

  • Different grain (one review vs one session)
  • Different update frequency (review data vs user profile)
  • Logically separate domain (review content vs device tracking)
  • Optional data (not all reviews have session data)

Add COLUMNS if:

  • Same grain (both describe the review)
  • Always present (every review has a timestamp)
  • Tightly coupled (review score and review text)

üí° Naming Convention:

  • fact_[event] ‚Üí Events/transactions
  • dim_[entity] ‚Üí Things/people/places
  • fact_[entity]_[feature_type] ‚Üí Derived features (e.g., fact_review_text_features)

üí° Practical Exercise: Identify Metrics

For each business question, identify: (1) The metric, (2) The dimension, (3) The tables needed

Question 1: "Which restaurants improved the most this year?"

  • Metric: Change in average rating
  • Dimension: Restaurant, time (year)
  • Tables: fact_review, dim_restaurant
-- SQL Pattern:
AVG(score) in 2024 - AVG(score) in 2023
GROUP BY restaurant_name

Question 2: "Do female users give higher ratings than male users?"

  • Metric: Average score
  • Dimension: Gender
  • Tables: fact_review, dim_users
-- SQL Pattern:
SELECT gender, AVG(score)
FROM fact_review
GROUP BY gender

Question 3: "What % of restaurants accept credit cards?"

  • Metric: Count of restaurants
  • Dimension: accepts_credit_cards (boolean)
  • Tables: fact_restaurant_attributes
-- SQL Pattern:
COUNT(*) / (SELECT COUNT(*) FROM fact_restaurant_attributes)
WHERE accepts_credit_cards = true

üìö Key Takeaways

  1. COUNT metrics: Things that happen multiple times
  2. SUM/AVG metrics: Numeric columns in fact tables
  3. Dimensions: How you slice the data (restaurant, user, time, geography)
  4. Same metric, different angles: The "slicing and dicing" pattern
  5. Missing metrics: Think about business decisions and dashboards
üí° Design Rationale - Why Each Decision Matters
üîë Why BIGINT for Fact Table IDs?

‚ùå INT (4 bytes)

Max: 2.1 billion rows

Problem: Walmart processes millions of transactions daily

Result: Will max out in ~3 years!

‚úÖ BIGINT (8 bytes)

Max: 9 quintillion rows

Benefit: Never run out

Cost: 4 bytes extra per row = $0.10/month for 1B rows

Verdict: Worth it! Migration costs thousands + downtime.
üìÖ Why date_key as INT (20241211)?
Aspect INT (20241211) DATE type
Performance ‚úÖ 10x faster comparisons ‚ùå Slower
Storage ‚úÖ 4 bytes ‚ùå 8 bytes (DATETIME)
Readable ‚úÖ 20241211 = Dec 11, 2024 ‚úÖ Human readable
Sortable ‚úÖ Numeric order = chronological ‚úÖ Sortable
üí∞ Why Pre-Calculate profit_amount?

Without Pre-Calculation (SLOW)

SELECT SUM(total_amount - cost_amount)
FROM fact_sales;
-- Calculates for EVERY row, EVERY time
-- Time: 45 seconds

With Pre-Calculation (FAST)

SELECT SUM(profit_amount)
FROM fact_sales;
-- Just sums, no calculation
-- Time: 2 seconds
Performance: 22x faster!
Why: Most queried metric - optimize for common case
üéØ Best Practices & Common Patterns
‚úÖ Golden Rules

1. Money = DECIMAL, ALWAYS!

‚ùå FLOAT
$19.99 ‚Üí $19.989999...
Sum of 1000: $19,989.99999
‚úÖ DECIMAL(10,2)
$19.99 ‚Üí $19.99
Sum of 1000: $19,990.00

2. Use Smallest Type That Fits

Data Type Why
Age TINYINT Max 255, perfect for age
Year SMALLINT -32K to 32K, enough for years
User ID INT Millions of users
Transaction ID BIGINT Billions of transactions

3. Surrogate Keys for Dimensions

-- Always use surrogate keys
product_key INT [pk, increment] -- Internal, stable
product_id VARCHAR(50) [unique] -- From source, can change

Why? Protects against source system changes, faster JOINs

4. Index Foreign Keys

Indexes {
  date_key [name: 'idx_date']
  product_key [name: 'idx_product']
  customer_key [name: 'idx_customer']
}

Why? Makes JOINs 100x faster!

Decision Reason Benefit
Surrogate keys Stability Source changes don't break warehouse
Integer date_key Performance 10x faster than date comparisons
DECIMAL for money Accuracy No rounding errors
Pre-calculated metrics Speed No computation during queries
Denormalized dimensions Simplicity Fewer JOINs
BIGINT for facts Scale Handles billions of rows

üìö Key Principles for Industry-Specific Models

  1. Understand Business Processes: Map the core operations of the industry.
  2. Identify Key Metrics: What drives success and what needs to be measured?
  3. Consider Regulatory Compliance: Integrate necessary attributes for reporting and auditing.
