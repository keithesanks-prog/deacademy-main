üìö What Are Materializations?

In dbt, a materialization is a strategy for persisting your transformed data in the warehouse. Think of it as answering the question: "How should dbt build this model in my database?"

Every dbt model needs a materialization strategy. The choice you make impacts:

  • Query Performance - How fast end-users can access the data
  • Build Time - How long dbt takes to refresh the model
  • Storage Costs - How much warehouse space you consume
  • Data Freshness - How up-to-date your data is
üí° Pro Tip: The default materialization is view, but you can (and should) customize this based on your specific use case!

What it does: Drops and recreates the entire table on every run.

2Ô∏è‚É£ View

What it does: Creates a database view - just stores the SQL query, not the actual data.

-- models/staging/stg_customers.sql
{{ config(materialized='view') }}

SELECT
id AS customer_id,
UPPER(name) AS customer_name,
email,
created_at
FROM {{ source('raw', 'customers') }}

When to Use

Staging models Simple transformations Infrequently queried models

Pros & Cons

‚úì No storage cost ‚úì Always fresh data ‚úì Fast to build
‚úó Slower query performance ‚úó Recomputes on every query ‚úó Can't add indexes

3Ô∏è‚É£ Incremental

What it does: Only processes new/changed records since the last run. The most powerful (and complex) materialization!

-- models/marts/fct_events.sql
{{ config(
materialized='incremental',
unique_key='event_id'
) }}

SELECT
event_id,
user_id,
event_timestamp,
event_type
FROM {{ ref('stg_events') }}

-- This filter only runs on incremental builds
{% if is_incremental() %}
WHERE event_timestamp > (SELECT MAX(event_timestamp) FROM {{ this }})
{% endif %}

When to Use

Large fact tables (millions+ rows) Event/log data Append-only datasets

Pros & Cons

‚úì Fast builds (only new data) ‚úì Fast queries (it's a table) ‚úì Cost-efficient for large data
‚úó More complex logic ‚úó Requires unique_key strategy ‚úó Can drift if logic changes

Incremental Strategies

dbt supports multiple strategies for handling duplicates:

  • append - Just add new rows (fastest, but allows duplicates)
  • merge - Update existing rows, insert new ones (default, most common)
  • delete+insert - Delete matching rows, then insert (Redshift/Postgres)
  • insert_overwrite - Partition-based replacement (BigQuery/Spark)

4Ô∏è‚É£ Ephemeral

What it does: Doesn't create anything in the database! Instead, dbt injects the SQL as a CTE in downstream models.

-- models/staging/stg_order_items_cleaned.sql
{{ config(materialized='ephemeral') }}

SELECT
order_id,
product_id,
quantity,
price * quantity AS line_total
FROM {{ source('raw', 'order_items') }}
WHERE quantity > 0

When you reference this model:

-- models/marts/fct_orders.sql
SELECT * FROM {{ ref('stg_order_items_cleaned') }}

dbt compiles it to:

WITH stg_order_items_cleaned AS (
SELECT
order_id,
product_id,
quantity,
price * quantity AS line_total
FROM raw.order_items
WHERE quantity > 0
)
SELECT * FROM stg_order_items_cleaned

When to Use

Intermediate transformations Reusable logic snippets Breaking up complex queries

Pros & Cons

‚úì Zero storage cost ‚úì Reduces model clutter ‚úì DRY principle
‚úó Can't query directly ‚úó Repeated computation if used multiple times ‚úó Harder to debug

üöÄ Advanced Patterns

Pattern 1: Hybrid Approach

Use different materializations at different layers:

  • Staging: Views (lightweight, always fresh)
  • Intermediate: Ephemeral (reduce clutter)
  • Marts: Tables or Incremental (fast queries)

Pattern 2: Incremental with Lookback Window

Process a few extra days to catch late-arriving data:

{% if is_incremental() %}
WHERE event_date >= DATEADD(day, -3, (SELECT MAX(event_date) FROM {{ this }}))
{% endif %}

Pattern 3: Project-Level Defaults

Set defaults in dbt_project.yml:

# dbt_project.yml
models:
my_project:
staging:
+materialized: view
marts:
+materialized: table
finance:
+materialized: incremental

‚ö†Ô∏è Common Mistakes to Avoid

‚ùå Mistake #1: Using table for everything

This wastes storage and slows down builds. Use views for staging!

‚ùå Mistake #2: Forgetting unique_key in incremental models

Without it, you'll get duplicate records. Always specify how to identify unique rows.

‚ùå Mistake #3: Not testing incremental logic

Run dbt run --full-refresh and compare results to catch bugs early!

‚ùå Mistake #4: Overusing ephemeral models

If a model is used by 10+ downstream models, make it a view instead to avoid repeated computation.

üí™ Practice Exercise

You're building a dbt project for an e-commerce company. Choose the right materialization for each model:

  1. stg_raw_orders - Cleans column names from raw orders table
    Answer: VIEW

    Simple transformation, feeds into other models, no need to store.

  2. fct_orders - Final fact table with 50M rows, queried by 20 dashboards
    Answer: INCREMENTAL

    Large dataset, frequently queried, append-only data.

  3. dim_products - Product dimension with 10K rows, updated daily
    Answer: TABLE

    Small enough for full rebuild, frequently joined in queries.

  4. int_order_calculations - Intermediate model calculating order totals, only used by fct_orders
    Answer: EPHEMERAL

    Intermediate logic, single downstream use, no need to materialize.

  5. rpt_daily_revenue - Aggregated report, 365 rows (one per day), queried hourly
    Answer: TABLE

    Small dataset, frequently queried, fast rebuild time is acceptable.

üéì Key Takeaways

  • View: Your default for staging and simple models
  • Table: For frequently-queried, small-to-medium datasets
  • Incremental: The power tool for massive fact tables
  • Ephemeral: For DRY code and intermediate logic
üéØ Golden Rule: Start simple (view/table), optimize later (incremental) when you have performance data. Premature optimization is the root of all evil!
